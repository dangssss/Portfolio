1. What v2 Fixes

V1 already had the main ML workflow. V2 raises the engineering bar: heavy jobs run in isolated Kubernetes pods, project code is packaged as a modular monolith, and monitoring is split between model-quality tables and infrastructure metrics.

  • Airflow uses KubernetesPodOperator for the main feature, EDA, and monthly model tasks.
  • Local deployment uses kind, Helm values, Docker images, Kubernetes secrets, and host-mounted churn data.
  • The accepted model bundle stores metrics, threshold, feature names, feature importance, model config, and pipeline config.

3. Dataset Preparation

The dataset pipeline is the core intellectual part of v2. It handles limited churn labels by combining business scope rules, temporal validation, prototype similarity, PU-style weighting, and label smoothing.

Temporal logic

  • 2-month prediction horizon.
  • Walk-forward window selection with minimum train windows.
  • Active and at-risk tiers using recency thresholds.
  • Train-only scaling to avoid leakage.

Label logic

  • Confirmed churners from CSKH files.
  • Reliable negatives from recent active customers.
  • Prototype-based pseudo churn via Mahalanobis similarity.
  • Sample weights and soft labels for uncertain groups.

6. Monthly Pipeline Contract

The monthly entry point python -m pipelines.monthly.monthly_v2_cli coordinates eight steps from dataset prep to export. If a new model fails the gate or loses to the prior bundle, the system can still score with the accepted bundle.
