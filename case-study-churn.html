1. System Goal

The system turns monthly VNPost customer and transaction data into an operational churn risk list. The output is designed for customer-care action: who is risky, how risky they are, and which behavioral signals explain the score.

  • Score active customers on a 2-month churn horizon.
  • Export high-risk customers into PostgreSQL and CSV-compatible tables.
  • Attach reason signals from SHAP where possible, with rule-based fallback when explanation artifacts are unavailable.

2. Airflow Chain

The real DAG chain is ds_churn_ingest -> ds_churn_features -> ds_churn_model_monthly. The model DAG runs python modeling/main.py run-monthly --horizon 2 --risk-threshold-pct 95.

3. ZIP Ingestion Design

The ingestion job is built around controlled full-refresh loads instead of ad hoc file appends. The goal is to make repeated monthly uploads auditable and recoverable.

Data controls

  • Unzip and discover source CSV files.
  • Validate expected headers and table schema.
  • Use PostgreSQL production schema after transformation.
  • Track successful loads in ingest.ingest_log.

Failure handling

  • Copy failed ZIP files to failed-data storage.
  • Preserve original input rather than deleting it.
  • Return structured status with rows, skip reason, success, and error.
  • Run post-ingest maintenance and analysis without failing the whole job.

4. Feature Generation

The feature layer creates static lifetime metrics and dynamic K-month behavior windows in PostgreSQL. The major engineering improvement in v1 is the smart recomputation strategy for sliding windows.

After18-20latest windows
Time45 -> 8mbenchmark note
Reduction96%less redundant work
  • Existing stable windows are kept; only new and latest incomplete windows are recomputed.
  • Window table names are generated dynamically from month ranges and K values.
  • Checkpoint and batch-insert logic reduce restart cost for long SQL feature jobs.

5. Monthly Modeling Pipeline

The monthly pipeline sweeps candidate lookback windows, compares a new candidate against the previous accepted model, retrains only when the candidate passes acceptance logic, and always scores active customers with the best available bundle.

6. Monitoring Tables

V1 already includes a production-monitoring layer in PostgreSQL, focused on model output quality rather than UI dashboards.

  • ml_monitor.run_log records each monthly run status, accepted flag, and notes.
  • score_drift stores active count, risk count, score quantiles, and risk-ratio anomalies.
  • feature_drift computes PSI and discrete KS by feature against the training profile.
  • backtest checks precision-in-list when future label windows become available.
